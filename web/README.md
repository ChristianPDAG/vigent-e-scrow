# Vigent Escrow Web App

This frontend is the user-facing flow for a Solana escrow where funds are locked by a depositor and released only after both the depositor and receiver confirm the in-person handoff.

## Ideal User Flow

### Depositor

The depositor is the user who creates and funds the escrow.

1. Open the app and connect a wallet using Phantom or another supported Solana wallet.
2. Go to the dashboard and create a new escrow.
3. Enter the receiver wallet address, amount, token type, expiry, title, and optional description.
4. Submit the escrow creation transaction in the wallet.
5. Fund the escrow from the escrow detail page.
6. Wait until the receiver is ready to complete the in-person exchange.
7. Join the release flow from the escrow detail page.
8. Scan the receiver's QR code or open the release link generated by the receiver.
9. Confirm the release transaction as depositor.
10. If the receiver already confirmed, the app finalizes the escrow and releases the funds.
11. If the receiver has not confirmed yet, wait for the receiver to confirm from their wallet.

Expected result: once both parties have confirmed, the funds move from the escrow vault to the receiver, minus the configured treasury fee.

### Receiver

The receiver is the user who receives the escrowed funds after the handoff is confirmed.

1. Open the app and connect the wallet address used as the escrow receiver.
2. Open the escrow detail page.
3. When the escrow is funded, enter the release page.
4. Generate the release QR code and session token.
5. Show the QR code to the depositor during the in-person exchange.
6. If the depositor scans and confirms first, keep the release page open or reload it if needed.
7. Confirm the release transaction as receiver.
8. If the depositor already confirmed, the app finalizes the escrow immediately after the receiver confirms.
9. For SOL escrows, the app uses WSOL internally and unwraps the receiver balance after finalization when the receiver completes the flow.

Expected result: the receiver sees the escrow marked as released and receives the funds in their wallet.

### When The Depositor Confirms First

This is a supported flow.

1. The receiver generates the release QR/session.
2. The depositor scans or opens the release link.
3. The depositor confirms first.
4. The escrow moves to `release_pending` with `depositorConfirmed = true` on-chain.
5. If the receiver reloads `/app/escrow/[id]/release`, the app reloads the active release session from the escrow account.
6. The receiver confirms.
7. The app finalizes the escrow because both confirmations are now present.

The receiver does not need to generate a new session after reload as long as the active release session has not expired.

### When The Receiver Confirms First

This is also supported.

1. The receiver generates the release QR/session.
2. The receiver confirms from the release page.
3. The escrow moves to `release_pending` with `receiverConfirmed = true` on-chain.
4. The depositor scans the QR code or opens the release link.
5. The depositor confirms.
6. The app finalizes the escrow because both confirmations are now present.

### QR / Session Token Flow

The QR code contains the release payload required to hydrate the release page on another device.

The payload includes:

- `escrowId`
- `sessionId`
- `token`
- `depositorWallet`
- `receiverWallet`
- `expiresAt`

If QR scanning is not available, the receiver can copy the full session token and share it manually. The scan page also supports pasting the payload or opening a release URL with query parameters.

### Reload And Recovery Behavior

Release sessions are stored on-chain through the escrow account, not only in browser memory.

If the release page is refreshed while the escrow is in `release_pending`, the app reconstructs the active session from the escrow account by reading:

- `releaseSessionHash`
- `sessionExpiresAt`
- `depositorReleased`
- `receiverReleased`

This allows either party to continue the release flow after a reload, as long as the release session is still active.

### Happy Path Summary

1. Depositor creates escrow.
2. Depositor funds escrow.
3. Receiver opens release page and generates QR/session.
4. Depositor scans QR and confirms.
5. Receiver confirms.
6. App finalizes escrow.
7. Receiver receives funds.

The order of the two confirmations can be reversed. Finalization only happens after both confirmations are present.

## Local Development

Install dependencies and run the web app from this directory:

```bash
pnpm install
pnpm dev
```

Useful checks:

```bash
pnpm tsc --noEmit
pnpm lint
```

Use PNPM for this project. Do not use NPM or Yarn.
